# 자가개선 검사 시스템 설계 — Active Learning + 재학습 루프

> 목표: 오늘 만든 검증 레이어(감사추적·검토큐·드리프트)를 한 단계 더 올려,
> "사람 검토 → 라벨 축적 → 재학습 → 합격 판정 → 모델 교체"가 도는 자가개선 시스템.
> 차별점: 코드가 아니라 **"각 결정을 왜 그렇게 했는가"(도메인 판단)** 가 핵심. 결정마다 이유를 남긴다.

---

## 전체 루프

```
        ┌─────────────────────────────────────────────────┐
        │                                                   │
   추론 ──→ 감사기록(audit) ──→ confidence 낮음? ──예──→ 검토 큐
        │         │                  │아니오               │
        │         │              자동 승인              사람이 정답 확정
        │         ↓                                        │
        │   드리프트 모니터링 ──── alert ───┐              ↓
        │                                    │        교정 라벨 축적
        │                                    ↓              │
        │                          [재학습 트리거] ←────────┘
        │                                    │   (라벨 N건 또는 drift alert)
        │                                    ↓
        │                          재학습(LoRA) → 후보 모델
        │                                    ↓
        │                   고정 평가셋으로 수용기준 평가
        │                                    ↓
        │                   기존보다 나으면? ──아니오──→ 폐기(현행 유지)
        │                          │예
        │                          ↓
        └──────────────  모델 교체(승격) + 레지스트리 기록 + (악화 시 롤백)
```

## 기존 코드와의 매핑 (이미 있는 것 / 새로 만들 것)

| 단계 | 상태 | 위치 |
|---|---|---|
| 추론 + 감사기록 | ✅ 있음 | `app/main.py`, `app/audit.py` |
| confidence 기반 검토 분류 | ✅ 있음 | `audit.decide_review` |
| 검토 큐 / 정답 확정 | ✅ 있음 | `/review/queue`, `/review/{id}/correct` |
| 드리프트 감지 | ✅ 있음 | `audit.drift_report` |
| **교정 라벨 → 학습 데이터셋 추출** | 🔨 새로 | `scripts/export_labels.py` |
| **재학습 트리거 판단** | 🔨 새로 | `scripts/retrain_trigger.py` |
| **후보 모델 합격 판정(gating)** | 🔨 새로 | `scripts/acceptance_eval.py` 확장 |
| **모델 레지스트리 / 승격·롤백** | 🔨 새로 | `app/registry.py` |

## 결정 지점 (← 지민종님이 도메인 근거로 정할 것)

각 항목: 선택지 / 내 추천 / 도메인 트레이드오프. 정한 이유를 `docs/decisions.md`에 한 줄씩 남긴다.

### D1. 재학습을 언제 트리거하나?
- 선택지: (a) 교정 라벨 N건 누적 (b) 드리프트 alert (c) 주기적(예: 주 1회) (d) 조합
- 추천: **(a) OR (b)** — 라벨 N건 모이거나 드리프트 경보 중 먼저 오는 것.
- 트레이드오프: 자주 = 최신성↑ 비용↑ / 드물게 = 안정 but 품질 저하 늦게 발견. 제조 라인 변경 빈도에 맞춰 N 결정.

### D2. 재학습 데이터를 어떻게 구성하나?
- 선택지: (a) 교정분만 (b) 원본 train + 교정분 (c) 원본 + 교정분 가중(oversample)
- 추천: **(c)** — 기존 지식 잊지 않게 원본 유지 + 새로 틀린 케이스는 가중.
- 트레이드오프: 교정분만 쓰면 catastrophic forgetting 위험. 가중 과하면 새 분포에 과적합.

### D3. 액티브러닝 샘플 선정 기준은?
- 선택지: (a) confidence 낮은 것 우선(uncertainty, 지금 검토큐가 이미 이것) (b) + 다양성 샘플링
- 추천: **(a) 유지** — 단순·효과적. 다양성은 나중 확장.
- 트레이드오프: uncertainty만 쓰면 특정 어려운 클래스에 쏠릴 수 있음(예: inclusion).

### D4. 후보 모델을 언제 "승격"하나? (가장 중요 — 안전 게이트)
- 선택지: (a) 정확도만 비교 (b) **수용기준(위험점수) 개선 AND 유형정확도 비퇴보**
- 추천: **(b)** — 고정된 golden test set에서, 위험점수 ≤ 현행 AND 유형정확도 ≥ 현행 − ε(예 0.01) 일 때만 승격.
- 트레이드오프: 너무 엄격하면 영원히 교체 안 됨 / 느슨하면 나쁜 모델 출고. 규제 관점에선 엄격 쪽이 안전.

### D5. 승격 후 악화되면?
- 선택지: (a) 수동 롤백 (b) 운영 드리프트/정확도 악화 alert 시 자동 롤백
- 추천: **이전 어댑터 보관 + 우선 수동 롤백**(자동은 추후). 레지스트리에 직전 버전 항상 유지.

### D6. 라벨 신뢰도는?
- 선택지: (a) 검토자 1인 (b) 복수 합의(다수결)
- 추천: **(a) + 검토자 기록**(이미 reviewer 컬럼 있음). 합의는 추후 확장 여지.
- 트레이드오프: 1인은 빠르나 오라벨 위험. 합의는 정확하나 비용↑.

### D7. 재학습 방식/비용은?
- 선택지: (a) 매번 base에서 LoRA 새로 (b) 직전 어댑터에서 이어학습
- 추천: **(a) base에서 LoRA 재학습** — 재현성↑, 64분이라 감당 가능. (b)는 드리프트 누적 위험.

### D8. 데이터 거버넌스(추적)는?
- 어떤 데이터 스냅샷으로 어떤 모델이 나왔는지 레지스트리에 기록(model_version → data 해시/건수 → 평가결과). ISO식 추적성 완성.
